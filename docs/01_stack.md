# Стек технологий

## Базовый стек

- `Python 3.12+` — основной язык проекта.
- `NumPy` — базовые численные операции и работа с массивами.
- `SciPy` — численные методы, в том числе решение ОДУ для динамического режима при необходимости.
- `FastAPI` — API-слой и сервисная архитектура.
- `Pydantic` — строгая валидация входных параметров и схем обмена.
- `Dash` — пользовательский интерфейс инженерной панели.
- `Plotly` — интерактивные графики и тренды.
- `SVG` + локальные `assets` Dash — базовый слой 2D-мнемосхемы установки.
- локальный JavaScript в `assets` и clientside callbacks Dash — быстрые визуальные обновления 2D-сцены.
- `pytest` — автоматические тесты модели и API.
- `Git` — фиксация этапов разработки.

## Рекомендуемые дополнения

- `pandas` — работа со сценариями, архивами расчетов и табличным экспортом.
- `Matplotlib` — статические рисунки для пояснительной записки.
- `Docker Compose` — воспроизводимый локальный запуск перед защитой.

## Опциональные технологии

- `OpenModelica` — дополнительный инженерный слой, если потребуется более формальная модель.
- `Streamlit` — запасной UI-путь, если сроки не позволят довести Dash-интерфейс.

## Почему выбран именно этот стек

- Он сохраняет проект code-first: модель реализуется автором, а не берется из внешней коробки.
- Он хорошо документирован и широко распространен.
- Он позволяет держать весь показ локально, без зависимости от облака.
- Он подходит под архитектуру «расчетное ядро + API + операторский экран + тесты».
- Он хорошо сочетается с профилем НПО «Каскад-ГРУП»: автоматизация, SCADA-логика, учебные стенды, работа с уставками и сигналами.

## Архитектурные решения, подтвержденные документацией

- `FastAPI` использовать как модульное приложение с `APIRouter`, чтобы не складывать все в один файл.
- `Pydantic` использовать на границе API и сервисов для диапазонов параметров и сериализации результатов.
- `Dash` использовать как реактивный UI с layout и callbacks, а не как набор отдельных скриптов.
- `Plotly` использовать как стандартный слой трендов и графиков.
- `SVG` использовать как основной путь для 2D-модели MVP, а не собирать мнемосхему из случайного набора HTML-блоков.
- `Dash`-assets и clientside callbacks использовать для визуального слоя там, где это ускоряет UI, но не влияет на расчетную корректность.
- `SciPy` включать в ядро только там, где действительно нужна динамика; для первой версии допускается стационарный расчет.

## Что не брать в базовый MVP

- Тяжелые CAD/CAE-пакеты как основной путь реализации.
- Прямую интеграцию с реальной SCADA/OPC-инфраструктурой предприятия.
- Облачные сервисы как обязательную часть демонстрации.
- Чрезмерно сложную физическую модель до подтверждения исходных данных и времени на валидацию.
- Полноценный 3D-движок как обязательную часть MVP.

## Правило по версиям

Точные версии пакетов не фиксируются в этом плановом пакете намеренно. Документация сверена 2026-04-04, но конкретные версии нужно закреплять уже на этапе bootstrap проекта в `pyproject.toml` после пробного запуска и smoke-test.
