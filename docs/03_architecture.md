# Архитектура проекта

## Базовый принцип

Модель должна быть реализована внутри проекта. Библиотеки используются как инструменты вычисления, API и визуализации, но не как замена собственной предметной модели.

## Целевая схема слоев

1. `simulation core`
   - формулы, состояния, переходы, ограничения, сценарии;
   - расчеты стационарного режима;
   - опционально динамика через `SciPy`.

2. `application services`
   - orchestration расчета;
   - запуск сценариев;
   - формирование предупреждений;
   - подготовка трендов и агрегатов.

3. `API layer`
   - `FastAPI`;
   - модульные `APIRouter`;
   - схемы запросов и ответов на `Pydantic`.

4. `UI layer`
   - `Dash` layout;
   - callbacks;
   - графики `Plotly`;
   - панели параметров, статусов и сценариев;
   - 2D SVG-мнемосхема;
   - подготовка адаптера для будущего 3D-viewer.

5. `supporting infrastructure`
   - конфигурация;
   - экспорт;
   - логирование;
   - тестирование;
   - локальный запуск и packaging.

## Базовые API-контракты

- `GET /health` — признак готовности приложения.
- `POST /simulation/run` — разовый запуск расчета.
- `POST /simulation/preview` — быстрый пересчет по измененным параметрам.
- `GET /scenarios` — список готовых сценариев.
- `POST /scenarios/{scenario_id}/run` — запуск типового режима.
- `GET /state` — текущее состояние модели.
- `GET /trends` — данные для графиков.
- `GET /visualization/state` — будущий контракт агрегированных сигналов для 2D/3D-представления.

## Архитектурные решения по документации

- FastAPI строить как bigger application: отдельный `main.py`, отдельные router-модули, зависимости и автоматическая документация `/docs`.
- Pydantic использовать для числовых диапазонов, чтобы не пропускать физически некорректные параметры на входе.
- Dash держать отдельно от ядра, а обновление экрана реализовывать через callbacks, привязанные к параметрам и результатам.
- Plotly использовать как единый графический слой, чтобы не смешивать UI-технологии без необходимости.
- 2D-мнемосхему строить как отдельный слой представления, не смешивая SVG-структуру с формулами модели.
- Закладывать общий словарь визуальных сигналов: состояние узлов, подписи, цвета, тревоги, направления потоков.
- Будущий 3D-viewer подключать как отдельный адаптер, использующий тот же словарь сигналов, а не прямой доступ к расчетному ядру.

## Рекомендуемая модель вычислений

### Вариант A. Базовый

- стационарный расчет;
- пересчет по событию изменения параметров;
- сценарии как наборы предустановок;
- 2D SVG-мнемосхема с реактивным обновлением узлов и потоков;
- этого достаточно для MVP и предзащиты.

### Вариант B. Усиленный

- стационарный плюс динамический режим;
- упрощенный контур регулирования;
- временные ряды;
- отдельный 3D-viewer, читающий те же визуальные сигналы, что и 2D;
- этот вариант включать только после валидации базовой версии.

## Рекомендуемая схема визуализации

### 2D-слой для MVP

- основа: SVG-файл в `assets` с `viewBox`, чтобы схема масштабировалась адаптивно;
- каждый узел установки получает стабильный `visual_id`;
- состояние узлов, подписи и тревоги собираются в отдельный view-model слой;
- быстрые визуальные обновления допускается выносить в clientside callbacks.

### Фундамент под 3D

- 3D не встраивается в расчетное ядро;
- 3D использует тот же `visual_id` и тот же набор сигналов, что и 2D;
- геометрия, материалы и camera presets должны лежать отдельно от расчета и отдельно от SVG;
- переключение 2D/3D должно происходить на уровне UI-адаптера, а не сервисов модели.

## Опциональный путь с OpenModelica

Если будет принято решение усиливать инженерную часть, OpenModelica нужно подключать как отдельный адаптер, а не перестраивать весь проект вокруг него. Базовая архитектура при этом остается той же:

- Python-сервис управляет сценарием;
- Python-UI отображает состояние;
- OpenModelica используется только как дополнительное расчетное ядро.

## Главный запрет для следующего этапа

Не начинать разработку с интерфейса. Сначала нужно утвердить параметры, уравнения, ограничения и целевой состав выходных показателей.

## P5. UI-эволюция: Concept-03 «Defense-Ready Digital Twin»

После M10–M18 архитектура UI расширяется отдельным parallel-пакетом
`src/app/ui/concept03/` (см. подробный план миграции:
`docs/concept03-defense-ready-digital-twin/12_migration_strategy.md`).

Принципы:

- расчётное ядро (`src/app/simulation`) **не меняется**;
- сервисы и API расширяются только аддитивно (новые поля/query-параметры
  с дефолтами);
- старый layout `src/app/ui/layout.py` остаётся как fallback (`?theme=legacy`);
- новый layout `src/app/ui/concept03/shell.py` живёт параллельно и
  активируется через `?theme=concept03` (`+ &defense=true` для Defense
  Day Variant).

Полный пакет архитектурной и UX-документации:
`docs/concept03-defense-ready-digital-twin/README.md`.
