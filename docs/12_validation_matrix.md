# Матрица валидации MVP

## Назначение

Этот документ фиксирует контрольные режимы текущей версии модели ПВУ и ожидаемые диапазоны результатов. Он нужен для двух задач:

- удерживать модель в воспроизводимом состоянии при доработках;
- иметь явный набор эталонных точек для фазы качества и подготовки к защите.

Исходные данные матрицы лежат в `data/validation/reference_points.json`, методические основания в `data/validation/reference_basis.json`, а agreed validation-contract в `data/validation/validation_agreement.json`. Они используются автотестом `tests/scenario/test_reference_matrix.py`, API endpoint'ами `GET /validation/matrix`, `GET /validation/agreement`, `GET /validation/basis`, встроенными dashboard-блоками `Validation Pack`, `Validation Agreement`, `Validation Basis` и живой инженерной сверкой `POST /validation/manual-check` / `Manual Check`.

## Контрольные режимы

| ID | Режим | Ожидаемый статус | Ключевые ожидания |
| --- | --- | --- | --- |
| `winter_supply_heating` | Зима, штатный нагрев | `normal` | приток держится около 21 °C, расход около 3323 м³/ч, тревог нет |
| `midseason_low_load` | Межсезонье, сниженная нагрузка | `normal` | низкая суммарная мощность, приток около 19 °C, тревог нет |
| `reduced_flow_alarm` | Снижение подачи | `alarm` | падение расхода до аварийного уровня, alarm `AIRFLOW_CRITICAL` |
| `dirty_filter_alarm` | Загрязнение фильтра | `alarm` | высокий перепад и сервисная тревога `FILTER_SERVICE_NOW` |
| `manual_mode_warning` | Ручной режим | `warning` | предупреждение `MANUAL_MODE` и пониженный расход |

## Что уже покрыто

- базовые режимы и сценарии из `data/scenarios/presets.json`;
- предупреждения по расходу, фильтру и ручному режиму;
- контроль температуры притока, расхода и суммарной мощности;
- regression-check на уровне API и расчётного сервиса;
- живая таблица сравнения «эталон -> факт» в UI и отдельный API-контракт для повторяемой проверки перед защитой.

## Сравнение с ручным расчётом

- `POST /validation/manual-check` строит пошаговую таблицу для активного набора параметров.
- В dashboard блок `Manual Check` показывает, как модель воспроизводит формулы по расходу, рекуперации, нагреву, фильтру, тепловому балансу и температуре помещения, а также явный допуск для каждого шага.
- Этот слой не заменяет будущую полевую калибровку, но делает расчёт объяснимым и повторяемым прямо во время демонстрации.

## Validation Agreement

- `GET /validation/agreement` и dashboard-блок `Validation Agreement` фиксируют, какие контрольные режимы и какие шаги `Manual Check` считаются согласованными для текущего generalized MVP-scope.
- В agreement snapshot хранятся центры допусковых окон, ожидаемые статусы/тревоги, step tolerances и границы применимости.
- Этот артефакт закрывает P1. Качество на уровне учебно-обобщённой ПВУ: дальнейшая работа переносится из общей validation-логики в сторону field calibration конкретного объекта.

## Методические основания

- `GET /validation/basis` и dashboard-блок `Validation Basis` показывают, на какие внешние материалы опираются шаги `Manual Check` и контрольные точки матрицы.
- Для каждого шага фиксируется тип основания: внешний источник, смешанное основание, производный шаг или внутреннее MVP-допущение.
- Этот слой не подменяет паспортную калибровку реальной установки, но убирает «чёрный ящик» между формулой, сценарием и литературной/методической опорой.

## Что остаётся после закрытия Phase 6

- при появлении паспортных данных конкретной установки заменить generalized control points на field-calibrated agreement;
- при необходимости добавить отдельные контрольные точки под новые сценарии защиты и зафиксировать их в `validation_agreement.json`;
- при появлении новых режимов добавлять их в JSON-матрицу и автотест, а не только в произвольные сценарии.
