# P0 Baseline

Дата фиксации: 2026-04-04.

## Назначение

Этот документ закрывает раздел `P0. Обязательно до старта кодирования` как рабочий baseline проекта. Он не подменяет внешнюю предметную экспертизу, но фиксирует минимальный согласованный scope, без которого нельзя поддерживать проект воспроизводимым.

Машиночитаемая версия того же baseline хранится в `config/p0_baseline.yaml` и доступна через `GET /project/baseline` вместе с одноимённым dashboard-блоком `P0 Baseline`.

## Что зафиксировано

- моделируется учебно-обобщенная приточная вентиляционная установка, а не паспортная копия конкретного агрегата предприятия;
- первая обязательная версия опирается на стационарный интерактивный расчет приточного тракта с упрощенным дискретным шагом по помещению;
- ПИД-регулятор и расширенная автоматика вынесены за пределы обязательного MVP;
- используются только синтетические или обезличенные данные.

| ID | Параметр | Единицы | Зачем обязателен |

| `heater_power_kw` | Доступная мощность нагревателя | кВт | Ограничение по догреву приточного воздуха |
| `filter_contamination` | Степень загрязнения фильтра | доля | Основа для перепада давления, падения расхода и тревог |
| `fan_speed_ratio` | Относительная скорость вентилятора | доля | Определяет фактический расход и мощность вентилятора |

## Минимальный набор сценариев защиты

| ID | Сценарий | Роль в защите |
| --- | --- | --- |
| `midseason` | Межсезонье | Базовый устойчивый режим и нормальный статус |
| `winter` | Зима | Показ рекуперации и нагревателя при холодном наружном воздухе |
| `reduced_flow` | Снижение подачи | Демонстрация аварийной реакции по расходу |
| `dirty_filter` | Загрязнение фильтра | Демонстрация сервисной тревоги и роста перепада давления |
| `manual_mode` | Ручной режим | Демонстрация warning-логики при operator override |

## Формат валидации модели

Принят четырехслойный формат, который уже реализован в коде:

1. `Validation Pack`
   Внутренняя regression-matrix по эталонным режимам из `data/validation/reference_points.json`, доступная в dashboard и через `GET /validation/matrix`.
2. `Validation Agreement`
   Зафиксированный validation-contract с control points и step tolerances в `data/validation/validation_agreement.json`, доступный через `GET /validation/agreement`.
3. `Manual Check`
   Пошаговое сравнение активного режима с ручным инженерным расчетом через `POST /validation/manual-check`.
4. `Validation Basis`
   Трассировка шагов и сценариев к внешним источникам и MVP-допущениям через `GET /validation/basis`.

## Что остается открытым, но уже не блокирует P0

- отдельная field calibration под паспортные данные конкретной установки не подменяется этим baseline;
- подтвержденный browser/WebGL profile не делает 3D обязательным: переход к обязательному WebGL после MVP должен оформляться как новая baseline-фиксация;
- появление подтвержденных паспортных данных конкретной установки должно оформляться как новая версия baseline, а не как тихая замена текущего scope.
