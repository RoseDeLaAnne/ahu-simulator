# Defense Q&A Pack

Дата: 2026-05-02.

Этот файл держать рядом с `docs/30_defense_presenter_script.md`. Ответы
короткие, чтобы их можно было использовать устно без чтения длинного текста.

## 1. Почему выбраны FastAPI и Dash?

FastAPI дает строгий и проверяемый API-контур: health-check, расчетные
endpoint'ы, exports, comparison и readiness. Dash удобен для локального
инженерного dashboard: можно быстро связать параметры, графики, таблицы и
callbacks без отдельного frontend-стека.

## 2. Как проверялась корректность модели?

Корректность проверялась несколькими слоями: unit-тесты расчетного ядра и
сервисов, integration-тесты API, scenario-тесты presets, ручная инженерная
сверка и browser/API smoke перед freeze. Финальный baseline: `198 passed`.

## 3. Почему модель учебно-обобщенная, а не паспортная копия установки?

Цель ВКР - показать расчетную структуру, режимы, статусы, визуализацию,
отчеты и сравнение сценариев. Паспортная цифровая копия требует точных данных
оборудования, телеметрии и отдельной валидации на объекте; это логичный
следующий инженерный этап.

## 4. Где хранятся runtime artifacts?

В режиме разработки runtime-артефакты пишутся в `artifacts`: exports,
comparison snapshots, event log, demo packages, simulation session и user
presets. В desktop/frozen режиме используется `%LOCALAPPDATA%\AhuSimulator`.

## 5. Чем полезен `scenario-report.v2`?

Это версионированный контракт отчета. Он объединяет metadata, секции, CSV, PDF,
manifest и проверочные сведения об артефактах. Благодаря версии контракт можно
расширять без неожиданного слома потребителей.

## 6. Зачем нужен Before/After Comparison?

Он отвечает на практический вопрос: что изменилось между двумя режимами.
Пользователь фиксирует `До` и `После`, сервис проверяет совместимость,
считает KPI/trend дельты и дает интерпретацию: улучшилось, ухудшилось или не
изменилось.

## 7. Почему системные presets locked?

Системные presets - это часть исходных данных проекта и тестового baseline.
Если их случайно менять во время показа, воспроизводимость пропадает. Поэтому
`winter`, `summer`, `peak_load` и другие системные presets read-only, а
пользовательские сценарии хранятся отдельно в runtime.

## 8. Что делать, если браузер или WebGL ведет себя нестабильно?

Не задерживаться на WebGL. Основной MVP-показ работает через 2D SVG, KPI,
тренды, Session v2, Reporting v2, Comparison, User Presets, API и сохраненные
screenshots/evidence. 3D/WebGL - необязательный слой.

## 9. Как воспроизвести результат на другой машине?

Поднять проект через `.\start.bat`, открыть фактический URL из консоли,
проверить `/health`, затем `/dashboard`. После этого выполнить короткий
click-pass из `docs/29_defense_handoff.md` или `docs/28_defense_freeze_note.md`.

## 10. Почему отчет включает manifest?

Manifest фиксирует состав пакета: пути, schema/version metadata и сведения для
проверки файлов. Это делает export не просто скачиванием PDF, а воспроизводимым
набором артефактов.

## 11. Как устроена связь между статусами и API?

Внутри API сохраняются технические значения `normal`, `warning`, `alarm` для
стабильности контрактов. В пользовательских слоях они отображаются как
`Норма`, `Риск`, `Авария` через единый status service.

## 12. Что будет, если comparison pair несовместима?

Сервис возвращает понятное объяснение и блокирует export. Это защищает от
некорректного сравнения режимов с разным шагом, горизонтом или временной
сеткой.

## 13. Почему runtime user presets не пишутся в `data/scenarios/presets.json`?

`data/scenarios/presets.json` - source-controlled baseline. Runtime presets -
пользовательские данные, поэтому они пишутся отдельно и могут быть очищены
после демонстрации без изменения исходников.

## 14. Какие главные доказательства готовности?

Full regression `198 passed`, `/health status=ok`, Phase H browser walkthrough,
Phase I handoff index, готовые CSV/PDF/manifest report/comparison packs,
Demo Bundle и screenshots в `artifacts/release-readiness/2026-05-02/`.

## 15. Какие дальнейшие улучшения после защиты?

Уточнение модели по паспортным данным конкретной установки, PID-регулятор,
подключение реальной телеметрии, расширенная валидация на объекте, развитие
3D/WebGL-слоя и packaging под целевой production-контур.
