# План реализации пакета M10-M14

Дата: 2026-04-18.

## Цель

Реализовать единый пакет пользовательских улучшений:

1. Расширение симуляции (`Старт`, `Пауза`, `Сброс`, шаги времени).
2. Отчетность по сценарию (`PDF/CSV`).
3. Сравнение прогонов `до/после`.
4. Цветные статусы `Норма/Риск/Авария`.
5. Пресеты сценариев (`Зима`, `Лето`, `Пик нагрузки`).

## Границы работ

### Входит в пакет

- Изменения в слое `services`, API-контрактах и dashboard для пяти функций.
- Обновление тестов (`unit`, `integration`, `scenario`) на критичные ветки поведения.
- Обновление артефактов и документации (`docs`, `artifacts/exports`, `scenario archive`).

### Не входит в пакет

- Реальная интеграция с внешним оборудованием и SCADA.
- Полноценная OPC-интеграция.
- Изменение validated Python-core формул вне задач пакета.

## Рабочие потоки

## Поток A. Временная симуляция

Ожидаемый результат:

- запуск/пауза/сброс прогона без перезагрузки;
- пошаговое обновление состояния и истории для графиков.

Шаги:

1. Зафиксировать модель `simulation session` (состояние, счетчик шага, история).
2. Добавить команды жизненного цикла: `start`, `pause`, `reset`, `tick`.
3. Провести поток данных в dashboard (кнопки + состояние).
4. Обновить тренды и KPI на основе истории шагов.
5. Добавить тесты переходов состояния сессии.

## Поток B. Цветные статусы

Ожидаемый результат:

- единые и объяснимые статусы `Норма/Риск/Авария` во всех представлениях.

Шаги:

1. Зафиксировать пороги статусов на уровне конфигурации.
2. Реализовать единый вычислитель статуса в сервисном слое.
3. Привязать отображение к карточкам, мнемосхеме и трендам.
4. Включить статусы в экспорт и сравнение прогонов.
5. Добавить unit-тесты на граничные значения.

## Поток C. Пресеты сценариев

Ожидаемый результат:

- готовые сценарии запускаются в один шаг и повторяемо дают ожидаемую картину.

Шаги:

1. Актуализировать `data/scenarios/presets.json` (добавить `summer`, `peak_load`).
2. Добавить короткие описания и целевые эффекты пресетов.
3. Добавить явное применение пресета в UI.
4. Добавить маркер источника параметров (`preset/manual`) в API-ответ.
5. Покрыть пресеты тестами диапазонов.

## Поток D. PDF/CSV отчетность

Ожидаемый результат:

- один и тот же прогон выгружается и в табличный CSV, и в инженерный PDF.

Шаги:

1. Зафиксировать единый контракт данных отчета.
2. Реализовать CSV-экспорт из контракта.
3. Реализовать PDF-экспорт из того же контракта.
4. Добавить manifest в каталог артефактов.
5. Добавить точки запуска в UI и API.
6. Добавить smoke-проверки сохранения и формата файлов.

## Поток E. Сравнение до/после

Ожидаемый результат:

- пользователь выбирает два прогона и получает таблицу дельт + визуальное сравнение.

Шаги:

1. Определить источник пар прогонов (active + archive).
2. Зафиксировать правила совместимости (шаг, длительность, набор метрик).
3. Реализовать расчет дельт по KPI и трендам.
4. Добавить экран сравнения в dashboard.
5. Добавить экспорт результата сравнения (`PDF/CSV`).
6. Добавить тесты на несовместимые пары.

## Фазовый план внедрения

## Фаза F1. Основание данных и контракты

- модель `simulation session`;
- контракт статусов;
- контракт отчета;
- контракт сравнения.

Выход из фазы:

- контракты зафиксированы и покрыты базовыми unit-тестами.

## Фаза F2. UI-управление и пресеты

- кнопки `Старт/Пауза/Сброс`;
- обновление истории и графиков;
- применение пресетов `Зима/Лето/Пик`.

Выход из фазы:

- живой прогон управляется из UI и воспроизводится через пресеты.

## Фаза F3. Экспорт и сравнение

- CSV/PDF экспорт одного прогона;
- режим `до/после` и экспорт сравнения.

Выход из фазы:

- пользователь получает артефакты для обсуждения и принятия решений.

## Фаза F4. Стабилизация

- доработка тестов;
- проверка граничных режимов;
- обновление user-facing документации.

Выход из фазы:

- пакет функций готов к демонстрации и повторяемому использованию.

## Риски и предохранители

- Риск: дрейф статусов между UI/API/export.
  - Предохранитель: единый вычислитель статуса и snapshot-тесты представления.
- Риск: несинхронность данных сравнения.
  - Предохранитель: явная валидация совместимости пары перед расчетом дельт.
- Риск: дублирование логики между PDF и CSV.
  - Предохранитель: единый контракт отчета и две независимые проекции из него.
- Риск: нестабильность UI при частых обновлениях.
  - Предохранитель: ограничение частоты тиков и перенос промежуточного состояния в `dcc.Store`.
