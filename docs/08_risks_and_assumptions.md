# Допущения, ограничения и риски

## Базовые допущения

- Модель реализуется автором проекта внутри Python-ядра.
- Для демонстрации используются синтетические или обезличенные данные.
- Основной показ выполняется локально, без обязательного доступа в интернет.
- Первая версия может быть стационарной, если этого достаточно для согласованного MVP.
- OpenModelica не является обязательной частью первой итерации.
- Обязательная визуализация MVP реализуется как 2D SVG-модель.
- Полноценная 3D-сцена не обязательна для MVP, но технический фундамент под нее должен быть подготовлен.

## Ограничения

- Без подтвержденных исходных данных нельзя обещать высокоточную цифровую копию реального объекта.
- Без согласованной методики валидации нельзя заявлять инженерную достоверность выше уровня учебной модели.
- Без согласования с вузом нельзя формально трактовать способ раскрытия использования ИИ в ВКР.
- Без разрешения предприятия нельзя использовать реальные конфиденциальные материалы.

## Основные риски

### Риск 1. Слишком широкий scope

Причина:
желание сразу сделать и сложную физику, и богатый UI, и интеграцию.

Снижение риска:
держать MVP в рамках «собственная модель + сценарии + графики + предупреждения + локальный запуск».

### Риск 2. Недостаток данных для валидации

Причина:
нет паспортных точек или подтвержденных параметров конкретной установки.

Снижение риска:
делать учебную обобщенную модель, сверять ее по ручным расчетам и согласовывать границы применимости.

### Риск 3. Потеря времени на OpenModelica

Причина:
специализированный инструмент повышает сложность и порог входа.

Снижение риска:
рассматривать OpenModelica только как phase-2 enhancement после готового Python MVP.

### Риск 4. Интерфейс окажется важнее модели

Причина:
визуальная часть проще и быстрее дает ощущение прогресса.

Снижение риска:
не переходить к полной сборке Dash-интерфейса, пока не готовы формулы, ограничения и эталонные режимы.

### Риск 5. Использование чувствительных данных

Причина:
попытка привязать работу к реальному предприятию слишком буквально.

Снижение риска:
держать в проекте только учебные, обезличенные или согласованные данные.

### Риск 6. Нестабильная демонстрация на защите

Причина:
зависимость от внешних сервисов, сетей или ручной настройки.

Снижение риска:
готовить автономный локальный сценарий запуска и резервный demo-script.

### Риск 7. 3D-визуализация съест сроки MVP

Причина:
попытка слишком рано перейти к тяжелой сцене, WebGL и детальной геометрии.

Снижение риска:
сначала довести выразительную 2D-модель и только затем подключать 3D как отдельный renderer поверх уже готовых visual bindings.

## Практические правила

- Любая новая функция сначала проходит проверку на вклад в защиту ВКР.
- Любая сложность, не влияющая на MVP, уходит в backlog.
- Любая интеграция с внешним ПО считается вторичной.
- Любое использование ИИ фиксируется как вспомогательное, а не определяющее инженерные выводы.
- Любая 3D-задача допускается только если не ломает сроки и целостность обязательной 2D-визуализации.
